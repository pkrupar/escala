    Quick tips

    • HTML classes are always written in camelCase, not in PascalCase nor with underscores.
    • Dashes - have specific meaning, do not use them for words separation. Use camelCase instead.

    Components explained

    Imagine we want a button. This button should be very clickable and pretty. However, we don't want to overwrite every <button> in our application. So instead of styling <button> directly like so button { color: #888 }, we create a separete .button class which will represent our new pretty button component.

    Huray! We have our own clickable button and we didn't overwrite default styles of <button>.

    Component driven development offers several benefits when reading and writing HTML and CSS:

    • It helps to distinguish between the classes for the root of the component, descendant elements, and modifications.
    • It keeps the specificity of selectors low.
    • It helps to decouple presentation semantics from document semantics.
    Syntax: .componentName

    Component modifiers explained

    Just realized we need outlined buttons too. Ok, you might be thinking: Yeah, I'll just create .buttonOutlined class and it's done. But is .buttonOutlined really a new component worth having separate class or just modified .button?

    Yes, it's just modified .button. Here's how we deal with modifying.

    To break it down we added .button--outlined to the main .button. Modifiers always start with name of the main component and two dashed, like so .button-- and follow with modifier name. From modifier's name should be clear what modifier do.

    Syntax: .componentName--modifierName

    Recap:

    • .button is main component
    • .button--primary is modifier for main .button component
    • .button--outline is another modifier for main .button component

    Component descendants explained

    A component descendant is a child node dependent on a component. It's responsible for styling on behalf of a parent component.

    Let code speak for itself:

    // Worst
<div id="nav">
  <div>
    <div><a href="#">Home</a></div>
    <div><a href="#">Invoices</a></div>
  </div>
</div>

    // Bad
<nav>
  <ul>
    <li><a href="#">Home</a></li>
    <li><a href="#">Invoices</a></li>
  </ul>
</nav>

    // Good
<div class="nav">
  <div class="nav-list">
    <div class="nav-listItem"><a href="#" class="nav-link">Home</a></div>
    <div class="nav-listItem"><a href="#" class="nav-link">Invoices</a></div>
  </div>
</div>

    // Best
<nav class="nav">
  <ul class="nav-list">
    <li class="nav-listItem"><a href="#" class="nav-link">Home</a></li>
    <li class="nav-listItem"><a href="#" class="nav-link">Invoices</a></li>
  </ul>
</nav>
    Syntax: .componentName-descendantName

    Component states

    Use is-stateName for state-based modifications of components. The state name must be camelCase. Never style these classes directly. They should always be used as an adjoining class.

    JS can add/remove these classes. This means that the same state names can be used in multiple contexts, but every component must define its own styles for the state (as they are scoped to the component).

    Checkout buttons docs for a lot of real world examples.

    Syntax: .componentName.is-stateName

    Utilities (helpers)

    Utilities can be applied directly to any element, multiple utilities can be used together and utilities can be used alongside component classes.

    Utilities exist because certain CSS properties and patterns are used frequently. For example: vertical and horizontal spacing, vertical alignment, text styling. Relying on utilities can help to reduce repetition and provide consistent implementations. 

    <div class="u-mb1"></div>
    Syntax: .u-utilityName

    JavaScript classes

    React makes this section redundent, but I'll leave it here anyway.

    Usage of JavaScript specific classes reduces the risk that changing the structure or theme of components will effect any required JavaScript functionality. These classes should never be user for styling.

    In practice this looks like this:

    <a href="/login" class="button button--primary js-login">Login</a>
    Syntax: .js-targetName
